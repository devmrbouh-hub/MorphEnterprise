# Скиллы на базе git — новая память AI-агентов

**Докладчик:** Константин Крестников (Сбер / GigaChat)  
**Фокус:** актуальные рабочие концепции, без исторической эволюции агентов.

**Оценка для MorphEnterprise:** [comparisons/krestnikov-vs-morphenterprise.md](../comparisons/krestnikov-vs-morphenterprise.md)

---

## TL;DR

Современный универсальный агент — это **harness** с небольшим набором базовых тулов. Расширение возможностей идёт не через сотни function calls / MCP-тулов в контексте, а через **скиллы**: папки с `SKILL.md`, скриптами и **данными** в git-репозитории. Скилл-репозиторий + автообслуживание (pruning, консолидация, CI) = **новая память агента**: контекст подгружается по требованию, а не заранее через RAG или жирный system prompt.

---

## 1. Harness — рабочая модель универсального агента

**Harness** (упряжка): LLM + ограниченный набор инструментов, которые «тянут» агента по пространству данных (файлы, репозитории, задачи).

### Ядро (~5–30 тулов, не больше ~50)

| Категория | Примеры |
|-----------|---------|
| Файловая система | чтение, запись, редактирование файлов |
| Shell | выполнение bash-команд |
| Поиск | grep / ripgrep по большим файлам |
| Внешний мир | web search |
| Оркестрация | todo-лист, субагенты |

Больше ~30–50 встроенных тулов — модель путается, контекст съедается описаниями функций.

### Из чего состоит хороший harness

1. **Короткий system prompt** — «используй инструменты, вот что делать / не делать»
2. **Run loop** — ReAct: задача → тулы → ответ, пока задача не решена
3. **Управление контекстом** — суммаризация, субагенты без передачи всего контекста
4. **Конвенции** — `AGENTS.md` / `CLAUDE.md`, директории скиллов, опционально MCP
5. **Устойчивость** — ретраи, обрывы сети, длинные сессии (часы)
6. **Интерфейс** — чат, CLI, мессенджер (OpenClaw)

### Два режима работы

| Режим | Суть |
|-------|------|
| **Interactive / man-on-the-loop** | человек не контролирует каждый шаг, а «присматривает»; агент уточняет при необходимости |
| **Headless / fire-and-forget** | задача из CLI или файла → агент работает до результата; встраивается в CI/CD и пайплайны |

**Примеры harness:** Claude Code, Codex, Cursor (частично), OpenClaw, Hermes.

## 2. Скиллы — лёгкая альтернатива тяжёлым тулам и MCP

**Скилл** = директория + джентльменское соглашение:

```
skill-name/
├── SKILL.md          # короткое + полное описание
├── scripts/          # bash, Python — исполняемые инструменты
└── data/             # данные, с которыми работает скилл
```

### Двухэтапная загрузка (ключевая идея)

| Этап | Что в контексте | Когда |
|------|-----------------|-------|
| 1 | **Короткое описание** скилла | всегда (или при индексации) |
| 2 | **Полный SKILL.md** + скрипты | только когда агент решил использовать скилл |

Это решает проблему MCP/тулов с многостраничными описаниями: у агента может быть 50–100+ скиллов, и контекста хватает.

### Скилл vs MCP (практическое правило)

| Используй **скиллы** | Используй **MCP** |
|----------------------|-------------------|
| Разные задачи, разные инструменты | Однотипный поток задач |
| >10–15 тулов / >50 инструментов суммарно | <10 узкоспециализированных тулов |
| Нужна гибкость, лёгкое создание (текст + скрипты) | Стабильный внешний сервис, один сценарий |

**Нюанс:** некоторые harness (например Claude Code) оборачивают MCP в skill-like обёртку — тогда MCP тоже получает ленивую загрузку.

## 3. Git-репозиторий как скилл: skill-first архитектура

```
skill-repo/
├── SKILL.md
├── scripts/
├── data/
└── .github/workflows/  # CI как backpressure
```

### Зачем git

- **Один скилл — много агентов:** Cursor днём, Hermes/OpenClaw в Telegram вечером — все работают с одними данными
- **Коллаборация:** push/pull, merge; современные harness умеют разрешать конфликты без человека
- **CI/CD:** скилл встраивается в пайплайны как обычный репозиторий
- **Обмен:** скилл можно отдать коллеге целиком (знания + код + данные)

### Обязательные практики

1. В `SKILL.md` — правило: **начало работы → `git pull`**, **конец → commit + push**
2. **CI как backpressure** — линтеры, тесты, валидация артефактов агента
3. Данные кладутся **в тот же репозиторий**, что и скилл — не отдельный RAG

## 4. Скиллы = новая память агента

| Старый подход | Подход со скиллами |
|---------------|-------------------|
| RAG по перепискам | данные лежат в skill-repo |
| Fat system prompt | короткие дескрипторы скиллов |
| Всё в контекст до старта | агент **сам решает**, что загрузить при активации скилла |

---

## 5. Hermes и Curator — самообслуживание скиллов

- **автосоздание:** если на задачу >5 tool calls — кандидат в новый скилл
- **Pruning:** 30 дней без использования → урезанный дескриптор; 90 дней → архив
- **Консолидация** (раз в 7 дней): мелкие скиллы объединяются в крупные

---

## 6. Агентные циклы (три уровня)

- **Inner loop** — ReAct до завершения задачи
- **Ralph Loop (outer)** — `while`: «продолжай задачу»; работа днями; CI как backpressure
- **Meta loop** — полный сброс + архив; исследование с разных сторон (Karpathy)

---

## 7. Рабочие кейсы из доклада

| Кейс | Суть |
|------|------|
| **ДНК-скилл** | ~100 ГБ секвенирования в skill-repo; multi-agent доступ |
| **Поездки** | Cursor + Hermes/Telegram на одном git-repo |
| **HR-пайплайн** | итеративный фидбек → скилл самоулучшается (~90% точности) |
| **AutoAgent** | outer loop: 1/89 → 11/89 задач за выходные |
| **Хакатоны** | backend на harness; headless для автоулучшения метрик |

---

## 8. Практические выводы (автор)

1. Не раздувать тулы — расширять harness скиллами
2. Данные рядом со скиллом в git
3. CI обязателен при автономной работе
4. Два режима harness: интерактив и headless
5. Три цикла: inner, Ralph, meta
6. Curator: pruning + консолидация
7. Ничего не фундаментально — смотреть на автономность разработчика
