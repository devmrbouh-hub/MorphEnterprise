# Принципы MorphEnterprise

Десять инвариантов системы. Каждый принцип проверяем при dogfooding.

---

## P0 — Минимально достаточная организация

**Формулировка:** Система не создаёт виртуальное предприятие по умолчанию. Форма работы определяется расчётом: объём работы, требуемые компетенции, risk, контекст и профили доступных моделей (включая Director). Организация добавляется только там, где одна модель не закрывает capability gap.

**Анти-паттерн:** На каждую задачу — Discovery, Planner, 5 Worker'ов и критик.

**Проверка при dogfooding:** Для 10 задач фиксировать sizing и фактическую достаточность. Опечатка в README не должна порождать организацию.

---

## P1 — Организация важнее агентов

**Формулировка:** Постоянны структура, компетенции и Workspace. Worker'ы временны и создаются под задачу.

**Анти-паттерн:** Заранее создать «Архитектора», «Программиста», «Тестировщика» как постоянных агентов.

**Проверка:** В Registry нет «должностей», только компетенции и ресурсы.

---

## P2 — Workspace — центр тяжести

**Формулировка:** Главный объект — не Worker, а Workspace. Состояние, решения, артефакты и task graph живут в Workspace.

**Анти-паттерн:** Контекст проекта хранится в истории чата Worker'а.

**Проверка:** Новая LLM-сессия за 15 минут понимает проект, читая только Workspace.

---

## P3 — Управление через адресные срезы контекста

**Формулировка:** Управляющие роли (Director, Planner, RuntimeCoordinator) не загружают весь контекст в одну модель. Они запрашивают **context slice** из Workspace по компетенции и этапу задачи.

**Анти-паттерн:** «Оркестратор знает весь код и всю историю».

**Проверка:** У каждого Worker в логе виден ограниченный slice, достаточный для его задачи.

---

## P4 — Компетенции, не должности

**Формулировка:** Система определяет, какие навыки нужны, и решает: объединить в одного Worker'а, разделить, привлечь человека или вызвать инструмент.

**Анти-паттерн:** Штатное расписание из фиксированных ролей.

**Проверка:** SizingDecision ссылается на competency_id из Registry, не на «роль архитектора».

---

## P5 — Workflow — результат планирования

**Формулировка:** Процесс не выбирается из каталога «если X → A, B, C». Task graph создаётся под конкретную mission.

**Анти-паттерн:** Жёсткие DAG-шаблоны на все случаи жизни.

**Проверка:** Две разные mission одного типа могут получить разный task graph.

---

## P6 — Адаптивность (resize, не только restart)

**Формулировка:** При тупике, росте объёма или падении confidence система меняет структуру (executor_count, planner_depth, модели), а не только перезапускает промпт.

**Анти-паттерн:** Бесконечный retry тем же составом.

**Проверка:** Есть лимит реорганизаций и явные триггеры resize; RuntimeCoordinator инициирует изменение.

---

## P7 — Качество — отдельный контур, масштабируемый по risk

**Формулировка:** QC (критик, страж, арбитр) включается пропорционально risk_tier, а не на каждый шаг.

**Анти-паттерн:** Три роли контроля на исправление опечатки.

**Проверка:** Таблица risk → qc_depth соблюдается; low risk = self-check.

---

## P8 — Обязательный Handoff

**Формулировка:** Перед завершением Worker фиксирует в Workspace решения, уроки и артефакты. Знания не остаются «в голове» модели.

**Анти-паттерн:** Worker исчезает без записи в Project Memory.

**Проверка:** Каждый завершённый work unit имеет handoff-record.

---

## P9 — Честная эскалация

**Формулировка:** При нехватке capability эскалация идёт по цепочке: слабее модель → сильнее модель → больше Worker'ов → человек. Director может эскалировать сам себя (weak → strong).

**Анти-паттерн:** Слабый Director принимает необратимые решения о сложных mission без second opinion.

**Проверка:** confidence < порога → эскалация задокументирована в SizingDecision.

---

## Сквозной пример: спецификация Workspace

| Принцип | Как проявляется |
|---------|-----------------|
| P0 | Director: executor_count=2, planner_depth=1 — не полная организация |
| P1 | Worker'ы «API design» и «review» исчезают после ADR |
| P2 | ADR пишется в Workspace, не в чат |
| P3 | Writer получает slice: концепция §6–7, без кода |
| P4 | Компетенции: system_design, api_design |
| P5 | Task graph: draft → review → finalize |
| P6 | Если review провален — resize (добавить critic) |
| P7 | risk=medium → один critic |
| P8 | Handoff: ADR + lessons |
| P9 | При низком confidence Director — strong second opinion |
